# Tailor IS 架构优化 Spec

## Why

原 Tailor IS 平台采用全量 Java SpringCloud 微服务架构，资源占用过大，非核心模块（社区、学堂）与核心模块（AI 制版、区块链存证、多商户交易）同等部署，导致初期运营阶段资源浪费严重、运维成本高。需通过 "分层轻量化、性能聚焦化、资源弹性化" 三大策略优化架构，在保留 "AI + 区块链 + 多商户商城" 核心壁垒的基础上，实现 "性能不降级、资源更高效、成本更可控"。

## What Changes

- **MODIFIED**: 后端架构从全量 SpringCloud 微服务改为 "核心 SpringCloud Alibaba 微服务集群 + 非核心 SpringBoot 单体/Serverless" 混合架构
- **MODIFIED**: 消息队列从单一 RabbitMQ 改为 "RabbitMQ（核心实时场景）+ RocketMQ（批量处理场景）" 分层
- **MODIFIED**: 缓存体系从单一 Redis 集群改为 "Redis Cluster（核心）+ Redis 单机（非核心）"
- **MODIFIED**: 数据库从 MySQL 主从改为 "MySQL 分库分表 + TiDB（核心高频写入表）"
- **MODIFIED**: 部署模式从全量 Docker 容器化改为 "核心 K8s 弹性伸缩 + 非核心 Serverless"
- **ADDED**: AI 模型本地轻量部署 + 云端高阶调用分层策略
- **ADDED**: 区块链存证批量上链 + 哈希预计算机制
- **MODIFIED**: 前端引入 SSR + 预渲染 + GraphQL 接口聚合
- **MODIFIED**: 监控体系从通用监控改为分层监控
- **ADDED**: 多端离线能力与弱网适配

## Impact

- **受影响系统**: tailor-is-gateway、所有后端微服务模块、部署运维体系、前端三端（PC/商家后台/移动端）
- **影响代码**: 网关路由配置、服务间通信机制、数据库分库分表迁移、缓存分层逻辑、消息队列双选型集成、前端渲染方式变更
- **非破坏性变更**: 保留现有 API 接口契约，业务功能不降级

## ADDED 需求

### Requirement: 混合架构模式

系统 SHALL 采用 "核心微服务集群 + 轻量服务 + 中台复用" 的混合架构，分为五层：

- **前端接入层**: PC 端、H5/小程序、商家后台 —— 多端统一渲染 + 边缘缓存
- **网关路由层**: 核心网关 + 非核心网关拆分 —— 动态路由 + 流量分级管控
- **核心服务层**: AI 制版、区块链存证、交易结算 —— SpringCloud 微服务集群
- **轻量服务层**: 社区、学堂、供应链对接 —— SpringBoot 单体 + 插件化
- **数据存储层**: 分库分表 + 缓存分层 + 对象存储 —— 冷热数据分离 + 读写分离

#### Scenario: 核心交易请求优先调度

- **WHEN** 网关接收到交易类请求
- **THEN** 核心网关优先路由至核心服务集群，保障高优先级响应

### Requirement: AI 智能制版性能优化

系统 SHALL 对 AI 制版模块实施计算资源分层：

- 常规版型生成使用本地轻量模型，降低接口调用延迟
- 特殊体型/爆款版型调用云端分布式计算
- 基础版型数据库预加载至 Redis，版型生成速度提升 40%+
- 批量任务与实时任务分离调度，非高峰时段批量生成热门版型缓存

#### Scenario: 本地模型生成常规版型

- **WHEN** 用户请求常规体型版型
- **THEN** 系统调用本地轻量模型生成，优先从 Redis 缓存读取基础版型

### Requirement: 区块链存证批量上链

系统 SHALL 优化存证流程：

- 原创作品上传时本地生成哈希值并缓存
- 累计 100 条后批量上链，减少链上交互次数
- AI 相似度比对模型本地部署，高疑似作品直接拦截
- 存证证书存储在 OSS，仅哈希值上链

#### Scenario: 批量上链触发

- **WHEN** 缓存的待上链记录达到 100 条
- **THEN** 系统执行批量上链操作，更新存证状态

### Requirement: 交易结算优化

系统 SHALL 优化多商户交易模块：

- 订单分库：高频商户订单单独分库，热门商品订单缓存至 Redis
- 结算分账非实时处理，每日低峰时段批量计算
- 支付回调异步处理，通过消息队列解耦

#### Scenario: 热点订单缓存命中

- **WHEN** 用户查询热门商品订单
- **THEN** 系统优先从 Redis 缓存读取，未命中再查数据库

### Requirement: 前端性能优化

系统 SHALL 实施前端渲染优化：

- PC 端首页/商品详情采用 SSR，移动端核心页面预渲染
- 商品详情/订单页采用 GraphQL 减少请求次数
- 核心交易/售后场景引入小程序原生组件
- 首屏加载时间降低 2s+

#### Scenario: 商品详情页 GraphQL 聚合

- **WHEN** 用户访问商品详情页
- **THEN** 通过 GraphQL 单次请求获取商品信息、评价、店铺信息，替代多次 HTTP 请求

### Requirement: Serverless 非核心服务

系统 SHALL 将社区、学堂等非核心服务迁移至 Serverless 架构：

- 按调用量计费，降低闲置资源占用 80%+
- 核心时段预留少量常驻资源，降低冷启动延迟
- 通过前端加载态优化冷启动体验

#### Scenario: 社区服务冷启动

- **WHEN** 用户访问社区页面且函数处于冷状态
- **THEN** 前端展示加载态，Serverless 函数启动后返回数据

### Requirement: 分层监控体系

系统 SHALL 实施分层监控：

- 核心服务全维度监控（性能指标、日志、链路追踪、告警）
- 非核心服务基础监控（可用性、基础资源指标）
- 资源占用监控与自动调优

#### Scenario: 核心服务异常告警

- **WHEN** 核心交易服务响应时间超过阈值
- **THEN** 监控系统立即告警，触发弹性扩容

### Requirement: 多端离线能力

系统 SHALL 在移动端提供离线能力：

- 离线工单编辑、离线商品浏览
- 弱网环境下缓存数据，恢复网络后自动同步

#### Scenario: 弱网下工单编辑

- **WHEN** 用户在弱网环境编辑售后工单
- **THEN** 工单数据本地缓存，网络恢复后自动提交同步

## MODIFIED 需求

### Requirement: 后端技术栈（原需求1技术栈选型）

**修改前**: SpringCloud（全量微服务）+ RabbitMQ（全场景）+ Redis（单一集群）+ MySQL 主从（全库）

**修改后**:
- 核心服务框架: SpringCloud Alibaba（轻量版），Nacos+Sentinel 替代原生组件，降低资源占用 30%+
- 消息队列: RabbitMQ（核心实时场景）+ RocketMQ（AI 制版批量场景）
- 缓存体系: Redis Cluster（交易/AI 数据高可用）+ Redis 单机（社区/学堂数据）
- 数据库: MySQL 分库分表 + TiDB（交易订单表高并发写入）

### Requirement: 前端技术栈（原需求1技术栈选型）

**修改前**: UniApp（全场景）+ 基础懒加载 + 常规 HTTP 请求

**修改后**:
- 跨端适配: UniApp + 小程序原生组件（交易/售后核心场景）
- 渲染优化: SSR（PC 端首页/商品详情）+ 预渲染（移动端核心页面）
- 接口交互: GraphQL（商品详情/订单页）+ HTTP（非核心场景）

### Requirement: 运维部署架构

**修改前**: 云服务器集群 + Docker 全量容器化 + CDN 常规加速 + 通用监控

**修改后**:
- 核心服务: K8s 轻量版 + 弹性伸缩
- 非核心服务: Serverless（函数计算）
- 静态资源: 边缘节点 CDN + 本地缓存
- 监控体系: 分层监控（核心全维度 + 非核心基础）

### Requirement: 性能优化策略（原需求6）

**修改前**: 数据库索引 + SQL 优化 + Redis 缓存 + 前端压缩 + CDN + 负载均衡 + 多级缓存

**修改后**（在原基础上增加）:
- AI 制版计算资源分层 + 算法预加载优化
- 区块链存证批量上链 + 哈希预计算
- 订单分库 + 热点订单缓存 + 非实时批量结算
- 前端 SSR + 预渲染 + GraphQL 接口聚合
- 数据同步分层（核心实时 + 非核心准实时 5 分钟）
- 多端离线能力 + 弱网适配

## REMOVED 需求

无删除需求，所有业务功能保留，仅架构模式优化。
