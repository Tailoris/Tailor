纸样定制”功能模块方案，将遵循 “从核心到外围，从基础到智能” 的迭代原则，拆解为四个清晰的阶段。

📜 项目总纲：纸样定制功能模块开发手册
1. 项目概述

名称：纸样定制功能模块（工厂版）

核心目标：在服装厂（需求方）与认证版师（服务方）之间，建立一套有平台担保的线上交易与协作流程。

核心价值：流程标准化、资金有担保、协作留痕迹、争议可仲裁。

用户角色：

工厂：发布需求、选择版师、支付款项、验收成果。

认证版师：浏览需求、投标报价、签订合同、交付作品。

平台管理员：审核订单、处理争议。

2. 技术栈与规范

前端：Next.js 14 (App Router), TypeScript, Tailwind CSS, 组件库（Ant Design / Shadcn/ui）。

后端：NestJS (或Next.js API Routes)， PostgreSQL/MySQL。

关键约定：

API响应格式：统一为 { code: number, data: any, message: string }。

错误处理：使用HTTP状态码和业务错误码。

类型定义：所有核心数据（订单、用户）必须使用TypeScript Interface定义。

🚀 第一阶段：核心交易流程MVP（1-2周）
目标：跑通“发布-投标-签约-付款”最简闭环，先解决“有无”问题。

任务1.1：扩展数据模型
指令：

在现有的数据库中，为支持定制订单功能，请创建或修改以下表结构。请给出SQL创建语句或Prisma Schema定义。

定制订单表 (custom_orders)：

id (主键), order_sn (订单号，唯一), factory_user_id (工厂用户ID), title (需求标题), description (详细描述), budget_range (预算范围，如 [5000, 10000]), status (状态，枚举值: 'bidding'-招标中, 'pending_contract'-待签约, 'in_progress'-进行中, 'completed'-已完成, 'cancelled'-已取消)。

投标表 (bids)：

id, order_id, designer_user_id (版师ID), quotation (报价金额), proposal (投标方案文本), status (状态: 'submitted', 'win', 'lose')。

请同时思考，如何关联现有的用户表(users)和版师店铺表(designer_profiles)。

任务1.2：工厂端-发布需求页
指令：

创建一个页面组件 app/custom-order/new/page.tsx。这是一个表单页，供工厂发布定制需求。

字段：

需求标题 (input)

详细描述 (textarea，支持富文本或Markdown)

预算范围（两个input，分别用于最小和最大金额）

期望交付日 (date picker)

附件上传 (file upload，用于上传设计草图/参考图)

逻辑：提交表单时，调用 POST /api/custom-orders，将数据保存到数据库，并将订单状态设为 'bidding'。成功后跳转到订单详情页。

UI：使用卡片式布局，表单垂直排列，提交按钮位于底部。

任务1.3：版师端-招标大厅与投标
指令：

创建招标大厅页面 app/bidding-hall/page.tsx。

功能：仅对designer_profiles表中状态为“已认证”的版师可见。

展示：列表形式展示所有 status='bidding' 的定制订单，显示标题、款式要求、预算范围、发布时间。

筛选：页面顶部提供简单的筛选（按预算区间、发布时间）。

创建投标弹窗/页面：

在招标大厅点击某个订单，进入投标页 app/bidding/[orderId]/bid。

页面预展示订单详情，下方是投标表单（报价input、方案阐述textarea）。

提交投标时，调用 POST /api/bids，创建一条投标记录，状态为 'submitted'。

任务1.4：工厂端-选择版师与合同
指令：

创建订单管理页 app/factory/orders/[id]/page.tsx。

工厂用户可查看自己发布的订单详情。

在 'bidding' 状态下，页面应列出所有投标，并显示版师头像、昵称、历史评分、报价和方案。提供一个【选择此版师】按钮。

实现选择逻辑：

点击【选择此版师】后，调用 POST /api/orders/[id]/select-bid。

此API需做三件事：a) 将订单状态更新为 'pending_contract'；b) 将中标投标状态设为 'win'，其他投标设为 'lose'；c) 向双方发送通知。

🔒 第二阶段：引入担保与协作（1-2周）
目标：嵌入信任基石，让交易安全、协作可追踪。

任务2.1：实现平台担保支付流程
指令：

修改订单模型，增加支付相关字段：

total_amount (合同总金额，由中标价确定)

payment_status (支付状态: 'pending', 'partially_paid', 'fully_paid')

payment_schedule (JSON字段，存储分期计划，如 [{stage: '签约', percentage: 30, status: 'paid'}, {stage: '初稿', percentage: 40, status: 'pending'}])

创建支付触发接口：

当工厂选择版师后，系统自动生成支付计划。

创建 POST /api/orders/[id]/create-payment 接口，调用第三方支付（如支付宝、微信）生成一笔金额为 total_amount * 30% 的支付订单，并将支付链接返回给前端。支付成功后，更新订单的 payment_status 和 payment_schedule。

在订单详情页集成支付组件：在 'pending_contract' 状态下，显示一个醒目的“支付30%定金”按钮，点击后调用上述接口并跳转支付。

任务2.2：建立纸样定制工作台（基础版）
指令：

创建一个内部协作页面 app/workspace/[orderId]/page.tsx，供工厂和版师使用。

功能模块（用Tab切换）：

聊天模块：集成一个简单的实时聊天（可用socket.io或定时轮询），所有消息存入数据库，并关联本orderId。

文件模块：提供一个文件上传区域，支持版师上传纸样文件（PRJ, DXF），工厂可下载。每次上传记录版本。

交付确认按钮：当版师上传“初稿”后，工厂在此页面可见一个【确认初稿，支付下一阶段款项】按钮，点击后触发支付流程。

任务2.3：基础仲裁申请入口
指令：

在项目工作台页面，添加一个“提起申诉”按钮。

点击后弹窗，表单包含：申诉类型下拉选择（“严重超时”、“质量严重不符”）、详细说明。

提交后，在后台创建一个仲裁记录，并将订单标记为“争议中”，同时通知平台管理员。

本阶段仅需实现申请入口和数据记录，人工后台处理即可。

⚙️ 第三阶段：流程自动化与体验优化（1-2周）
目标：让系统更智能、更规范。

任务3.1：电子合同签署流程
指令：

设计合同模板：创建一个文本模板，包含 {orderSn}, {factoryName}, {designerName}, {amount}, {deliveryDate} 等占位符。

创建签约页面：在工厂支付定金后，自动跳转到签约页。页面展示填充后的合同内容，底部有“我已阅读并同意”复选框和【签署合同】按钮。

实现签署逻辑：点击签署按钮，记录签署时间、用户IP，并更新订单至 'in_progress' 状态。同时，生成一份PDF合同存档。

任务3.2：投标智能参考（基础版）
指令：

在版师投标页面（app/bidding/[orderId]/bid）的报价输入框旁，添加一个“查看价格参考”链接。

点击后，调用 GET /api/bidding/price-reference?category=连衣裙。

后端接口实现：根据当前订单的标题或标签，从历史完成的订单中，查询同类目的最终成交价，计算平均值、最低值、最高值，返回给前端。

前端以简洁的文字形式展示：“同类历史订单平均成交价：¥4,500（区间：¥3,000 - ¥6,000）”。

🧠 第四阶段：高级功能与智能化（后续迭代）
目标：构建核心竞争力和护城河。

任务4.1：高级仲裁案卷系统
指令：

为管理员开发一个仲裁后台页面 app/admin/arbitration/[caseId]/page.tsx。

页面布局：左侧为订单完整时间线（整合聊天、支付、文件上传事件），右侧为证据查看区（可查看工作台所有聊天记录和文件）。

裁决操作：管理员可填写裁决意见，并选择裁决结果（如“支持工厂，退款50%”）。提交裁决后，系统应能自动执行退款操作。

任务4.2：版师竞争力分析雷达图
指令：

在版师投标时，为其生成一个竞争力分析报告。

后端计算：编写一个函数 calculateMatchScore(orderId, designerId)，从四个维度评分（历史作品风格匹配度、客户评价、准时交付率、报价竞争力），每个维度返回一个0-1的分数。

前端可视化：使用 recharts 或 victory 库，将四个维度的分数绘制成雷达图，展示在投标表单侧边，并给出文字建议（如“您的风格匹配度很高，可适当突出此优势”）。
