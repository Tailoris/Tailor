## 一、自动化测试策略

### 1.1 测试分层与覆盖率目标

#### 单元测试

- **后端(Java)**: 覆盖率目标 70%+（行覆盖），使用 JUnit 5 + Mockito
  - 核心服务(user/order/payment/copyright): 80%+
  - 辅助服务(product/merchant/marketing): 60%+
  - 工具类(common): 90%+
  - 当前状态: 约10%（参考 PROJECT-COMPREHENSIVE-AUDIT-FINAL.md）
  - 重点补充: PaymentService/EscrowService 核心测试、PatternService 测试、SysUserServiceImpl 状态逻辑测试、CopyrightServiceImpl 加密测试

- **前端(TypeScript)**: 覆盖率目标 60%+（行覆盖），使用 Vitest + Vue Test Utils
  - API 层(api/*.ts): 80%+
  - 工具类(utils/*.ts): 90%+
  - Store 层(store/*.ts): 70%+
  - 组件层(components/*.vue): 50%+
  - 当前状态: 仅 e2e-tests 存在，单元测试缺失
  - 重点补充: crypto.ts 加密测试、validate.ts 校验测试、request.ts 拦截器测试、store/user.ts 状态管理测试

#### 集成测试

- **后端**: 使用 Spring Boot Test + TestContainers，覆盖核心业务流程
  - 用户注册 → 登录 → 刷新 Token 全流程
  - 商品浏览 → 加购 → 下单 → 支付 → 回调全流程
  - 版权作品上传 → 相似度比对 → 存证上链全流程
  - AI 制版请求 → 任务调度 → 结果返回全流程

- **前端**: 使用 MSW(Mock Service Worker) 模拟后端接口，覆盖组件交互
  - 商品列表筛选/排序/分页
  - 购物车增删改
  - 订单创建/支付
  - 登录/注册/Token 刷新

#### E2E 测试

- **工具**: Playwright（已有 e2e-tests 项目）
- **覆盖率目标**: 关键用户旅程 100% 覆盖
- **已有测试**: auth.spec.ts、checkout-flow.spec.ts、homepage.spec.ts、product.spec.ts、accessibility.spec.ts
- **需补充**: 社区发帖流程、商家入驻流程、版权存证流程、订单售后流程、多店铺切换流程
- **执行频率**: 每次 PR 合并到 main 分支时执行

### 1.2 测试执行策略

- **本地开发**: 开发者提交前运行单元测试（快速反馈）
- **CI Pipeline**: PR 提交时运行单元测试 + 集成测试（质量门禁）
- **CD Pipeline**: 部署前运行 E2E 测试（发布门禁）
- **定期回归**: 每日夜间运行全量 E2E 测试（回归保障）

## 二、代码审查流程规范

### 2.1 PR 模板规范

更新 `.github/PULL_REQUEST_TEMPLATE.md`，包含以下审查清单：

#### 提交者自查清单

- [ ] 代码已通过本地单元测试
- [ ] 代码已通过 Checkstyle/PMD/ESLint 静态分析
- [ ] 未引入新的 `any` 类型（前端）/ 未引入新的魔法数字（后端）
- [ ] 未硬编码任何密码、密钥、Token
- [ ] 新增功能已编写单元测试
- [ ] Breaking Change 已在描述中标注
- [ ] 相关文档已更新

#### 审查者检查清单

- [ ] **安全审查**: 输入校验是否完整、认证授权是否正确、敏感数据是否加密、是否有 SQL 注入/XSS 风险
- [ ] **性能审查**: 是否存在 N+1 查询、是否有不必要的同步操作、缓存使用是否合理
- [ ] **逻辑审查**: 边界条件是否处理、异常处理是否完整、事务边界是否正确
- [ ] **规范审查**: 命名是否符合规范、代码结构是否清晰、是否有重复代码
- [ ] **测试审查**: 测试覆盖是否充分、测试用例是否有效、边界场景是否覆盖

### 2.2 审查规则

1. **强制审查**: 所有 PR 至少 1 名审查者批准方可合并
2. **双人审查**: 核心模块(user/order/payment/copyright) PR 需 2 名审查者批准
3. **安全审查**: 涉及认证/支付/加密的 PR 需安全负责人审查
4. **自动检查**: CI 通过是 PR 可合并的前提条件
5. **大小限制**: 单个 PR 改动不超过 500 行（不含自动生成代码），超过需拆分

### 2.3 审查时效

- 普通 PR: 24 小时内完成审查
- 紧急修复 PR: 4 小时内完成审查
- 安全修复 PR: 2 小时内完成审查

## 三、CI/CD Pipeline 质量门禁设置

      配置: tailor-is/checkstyle.xml

      必须通过: true
      配置: tailor-is/pmd/pmd-ruleset.xml
      失败动作: 阻断PR合并
    SpotBugs:
      必须通过: true
      失败动作: 阻断PR合并

    gitleaks(密钥扫描):
      必须通过: true
      失败动作: 阻断PR合并

## 四、定期质量评估与改进机制

### 4.1 周度质量报告

- **生成时间**: 每周一自动生成
- **报告内容**:
  - 本周 PR 合并数、审查平均时长
  - 单元测试覆盖率变化趋势
  - 静态分析问题数变化趋势
  - 安全扫描发现的新漏洞数
  - 生产环境告警数与处理时长
  - 技术债务问题数变化（新增/关闭/净增）
- **报告工具**: GitHub Actions + SonarQube + Prometheus 数据聚合
- **分发方式**: 自动发送至团队群聊/邮件

### 4.2 月度技术债务评估

- **评估时间**: 每月最后一个工作日
- **评估内容**:
  - 技术债务问题总数与按严重级别分布
  - 本月新增技术债务 vs 本月清偿技术债务
  - 高风险技术债务清单（P0/P1 级别未修复项）
  - 技术债务清偿率目标达成情况
  - 下月清偿计划
- **参与人员**: 技术负责人 + 各模块负责人

### 4.3 季度架构评审

- **评审时间**: 每季度末
- **评审内容**:
  - 架构文档与实际实现的一致性
  - 五层架构（前端接入/网关路由/核心服务/轻量服务/数据存储）健康度
  - 核心模块性能指标对照架构文档要求（AI 制版速度、区块链存证效率、订单处理峰值）
  - 安全合规评分变化趋势
  - 架构优化建议
- **参与人员**: 架构师 + 技术负责人 + 安全负责人
- **产出物**: 架构评审报告 + 优化建议清单

### 4.4 持续改进机制

1. **问题复盘**: 每次 P0 级生产事故后 48 小时内召开复盘会，产出根因分析和改进措施
2. **最佳实践沉淀**: 将代码审查中发现的通用问题沉淀为编码规范更新
3. **培训计划**: 每季度组织 1 次安全编码培训、1 次性能优化培训
4. **工具链优化**: 每半年评估 CI/CD 工具链效率，优化构建速度和反馈时效

## 五、验收标准

### 5.1 功能完整性验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 用户认证 | 注册/登录/刷新 Token/微信登录功能正常，正常用户可登录、禁用用户被拦截 | E2E 测试 + 手动验证 |
| 商品交易 | 商品浏览/加购/下单/支付/回调/结算全流程正常，资金入账正确 | E2E 测试 + 对账验证 |
| 版权存证 | 作品上传/相似度比对/高风险拦截/存证上链全流程正常，敏感数据加密存储 | E2E 测试 + 安全扫描 |
| AI 制版 | 制版请求/任务调度/结果返回流程正常（注：AI 推理引擎为独立里程碑） | 集成测试 |
| 多商户 | 商家入驻/多店铺切换/商品管理/订单管理功能正常 | E2E 测试 |
| 社区学堂 | 社区发帖/评论/学堂课程浏览功能正常 | E2E 测试 |
| 核心功能无缺失 | 对照架构文档五层架构，所有已实现模块功能完整 | 功能清单核对 |

### 5.2 性能指标验收（对照架构文档要求）

| 性能指标 | 架构文档要求 | 验证方式 |
|----------|-------------|----------|
| AI 制版生成速度 | 提升 40%+ | JMeter 压测(tailor-is/deploy/performance-test/ai-pattern-test.jmx) |
| 区块链存证效率 | 提升 50%+ | JMeter 压测(tailor-is/deploy/performance-test/blockchain-test.jmx) |
| 交易订单处理峰值 | 提升 50%+ | JMeter 压测(tailor-is/deploy/performance-test/trading-test.jmx) |
| 首屏加载时间 | 降低 2s+ | Lighthouse 审计 |
| 资源利用率 | 提升 40%+ | Prometheus 监控数据 |
| 运维成本 | 降低 50%+ | 资源占用对比 |

### 5.3 安全指标验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 安全漏洞修复率 | 100%（Critical+High 级） | Trivy + gitleaks + ZAP 扫描 |
| 生产凭据不入库 | 代码库中无任何真实密码/密钥 | gitleaks 全量扫描 |
| 数据加密合规 | 敏感数据使用 AES-256-GCM 加密，密钥从环境变量读取 | 代码审查 + 配置检查 |
| 认证授权正确 | 用户状态逻辑正确，网关鉴权链路完整 | 渗透测试 + 代码审查 |
| 部署安全基线 | Actuator 收敛、DB SSL、端口不暴露、K8s 安全上下文配置 | 安全配置扫描 |
| CI/CD 门禁生效 | 所有 PR 通过质量门禁方可合并 | CI 配置检查 |

### 5.4 用户体验指标验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 首屏加载时间 | <2s（PC 端 SSR） | Lighthouse 审计 |
| 接口响应时间 | P95 <500ms（核心接口） | APM 监控 |
| 页面交互响应 | <100ms（点击到反馈） | 性能埋点 |

| 无障碍访问 | WCAG 2.1 AA 标准 | axe-core 扫描 |

### 5.5 代码质量指标验收

| 验收项 | 验收标准 | 验证方式 |
|--------|----------|----------|
| 后端单元测试覆盖率 | 70%+（行覆盖） | JaCoCo 报告 |
| 前端单元测试覆盖率 | 60%+（行覆盖） | Vitest 覆盖率报告 |
| 静态分析问题数 | 0 个 Critical/High | SonarQube 报告 |
| 代码重复率 | <3% | SonarQube 报告 |
| TypeScript any 使用 | 0 处新增 | TypeScript 编译 + ESLint |
| 魔法数字使用 | 0 处新增（后端） | PMD 检查 |

### 5.6 验收流程

1. **自验**: 开发团队按上述标准自验，产出自验报告
2. **安全验收**: 安全团队执行 Trivy/gitleaks/ZAP 扫描，确认漏洞修复率 100%
3. **性能验收**: QA 团队执行 JMeter 压测，确认性能指标达标
4. **功能验收**: 产品团队对照需求文档验证功能完整性
5. **最终验收**: 技术负责人审核所有验收报告，签署验收结论
