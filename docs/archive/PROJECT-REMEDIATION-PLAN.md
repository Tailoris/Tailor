## 一、问题优先级分类标准及修复顺序建议

### 1.1 P0 - Critical（立即修复，24-48小时内）

**定义**：影响系统安全、资金正确性、核心业务可用性的严重问题，可能导致数据泄露、资金损失、服务中断。

| 问题类别 | 数量 | 问题清单 |
|----------|------|----------|
| 后端逻辑缺陷 Critical | 6项 | 用户状态反转×4、支付 merchantId 错误、版权高风险未拦截 |
| 后端安全漏洞 Critical | 5项 | AES 密钥硬编码、AES-ECB 模式、加密失败返回明文、K8s Secret 密码、沙箱控制器暴露 |
| 前端安全漏洞 Critical | 3项 | XOR 伪加密、platform-admin 无防护、refresh_token 明文 |
| 部署安全 Critical | 3项 | 生产密码入库、TLS 私钥入库、K8s 监控 namespace 不匹配 |
| **P0 合计** | **17项** | - |

### 1.2 P1 - High（本阶段修复，1-2周内）

**定义**：影响系统性能、并发安全、部署安全基线的问题，存在较高风险但非立即致命。

| 问题类别 | 数量 | 问题清单 |
|----------|------|----------|
| 后端 High | 9项 | 订单取消未释放库存、支付缓存无效、N+1 查询×3、事务并发×2、网关头未签名、输入校验缺失 |
| 前端 High | 4项 | mobile-app 弱加密、CSRF 可读、GraphQL 无鉴权、SSR 缺安全头 |
| 部署 High | 11项 | DB 禁 SSL、端口暴露、Actuator 全暴露、无 securityContext、无 PDB、无 NetworkPolicy、无速率限制、Alertmanager 未连接、监控覆盖不足、无安全扫描、测试不阻断 |
| **P1 合计** | **24项** | - |

### 1.3 P2 - Medium（下阶段修复，2-4周内）

**定义**：编码规范、类型定义、性能优化、逻辑缺陷等问题，影响代码可维护性和用户体验。

| 问题类别 | 数量 | 问题清单 |
|----------|------|----------|
| 后端 Medium | 15项 | 魔法数字、手动 JSON、空 catch、Lambda、缓存无效、配置重复等 |
| 前端 Medium | 9项 | `request<any,T>` 类型、公共类型重复、N+1 查询、Redis KEYS、状态不持久、逻辑分支冗余等 |
| 部署 Medium | 13项 | host 网络、Grafana 默认密码、CORS、CSP、保留期短等 |
| **P2 合计** | **37项** | - |

### 1.4 P3 - Low（优化阶段，1-2月内）

**定义**：文档缺失、代码风格、轻微优化等问题，不影响功能但影响开发体验。

| 问题类别 | 数量 | 问题清单 |
|----------|------|----------|
| 后端 Low | 6项 | 废弃代码、文档 TODO、测试覆盖不足等 |
| 前端 Low | 7项 | UUID 弱随机、命名不一致、密码策略弱、文档缺失等 |
| 部署 Low | 1项 | 索引覆盖良好（正面发现） |
| **P3 合计** | **14项**（含1项正面发现） | - |

### 1.5 修复顺序建议

按风险等级与依赖关系，分八批次推进修复：

| 批次 | 主题 | 修复内容 | 时间窗口 |
|------|------|----------|----------|
| 第一批 | P0 安全凭据 | 立即轮换所有入库的生产凭据，从 Git 历史清除，修复 K8s 监控 namespace | 24-48 小时 |
| 第二批 | P0 逻辑缺陷 | 修复用户状态反转、支付 merchantId、版权加密、订单库存释放 | 3-5 天 |
| 第三批 | P0 前端安全 | 移除 XOR 伪加密，加固 platform-admin，修复 refresh_token 存储 | 5-7 天 |
| 第四批 | P1 部署安全 | Actuator 收敛、DB SSL、端口收敛、K8s securityContext、Nginx 速率限制 | 第 2 周 |
| 第五批 | P1 性能并发 | N+1 查询优化、事务边界修复、DataLoader 引入 | 第 2-3 周 |
| 第六批 | P1 CI/CD | 移除 continue-on-error、添加安全扫描、质量门禁生效 | 第 3 周 |
| 第七批 | P2 规范类型 | 枚举替换、类型统一、JSON 拼接修复、前端类型重构 | 第 4-6 周 |
| 第八批 | P3 文档长效 | README 补全、长效机制建设、验收 | 第 7-8 周 |

## 三、代码重构方案

### 3.1 后端重构

| 序号 | 重构主题 | 重构内容 | 参考基线 |
|------|----------|----------|----------|
| 1 | 状态字段枚举化 | 将 `SysUser.status`、`OrderInfo.status`、`PaymentRecord.payStatus` 等 Integer 字段重构为枚举类型 | 已有的 `ProductStatusEnum` 和 `AuditStatusEnum` 模式 |
| 2 | 加密服务统一 | `CopyrightServiceImpl` 的 AES 加密迁移至项目已有的 `AesGcmCrypto`，移除手动 `Cipher.getInstance` 调用 | 项目既有 `AesGcmCrypto` 实现 |
| 3 | JSON 构建统一 | `CopyrightServiceImpl` 和 `PatternServiceImpl` 中手动拼接 JSON 替换为 `ObjectMapper` | Spring Boot 默认 Jackson |
| 4 | N+1 查询消除 | `PatternCacheLoader` 使用 DISTINCT 查询；`SysUserServiceImpl.buildUserInfo` 改为批量查询；`OrderInfoMapper` 评估 JOIN 替代嵌套查询 | JPA/MyBatis 性能最佳实践 |
| 5 | 事务边界修正 | `PaymentServiceImpl.payCallback` 修正 Redis 锁与事务生命周期一致性；`EscrowServiceImpl.deposit` 使用 `INSERT ON DUPLICATE KEY` | 分布式锁与事务一致性原则 |

### 3.2 前端重构

| 序号 | 重构主题 | 重构内容 | 参考基线 |
|------|----------|----------|----------|
| 1 | API 封装统一 | pc-mall 和 merchant-admin 的 `request<any, T>` 模式统一为 mobile-app 的 `post<T>(url, data)` 模式，消除 105 处 any | mobile-app 既有实现 |
| 2 | 公共类型抽取 | 在 `shared/` 目录创建 `types/` 文件夹，抽取 `ApiResponse`/`PageResponse`/`User` 等公共类型，消除三处重复定义 | TypeScript 类型复用 |
| 3 | 加密工具统一 | 移除 pc-mall 和 merchant-admin 中重复的 `crypto.ts` XOR 实现，统一使用 Web Crypto API | W3C Web Crypto API 标准 |
| 4 | GraphQL 优化 | 引入 DataLoader 解决 `Product.category` N+1 查询，Redis `KEYS` 替换为 `SCAN` | DataLoader 批量加载模式 |
| 5 | 状态管理完善 | merchant-admin 的 `currentShopId` 持久化，`fetchUserInfo` 失败跳转登录页 | 前端状态持久化最佳实践 |

### 3.3 部署重构

| 序号 | 重构主题 | 重构内容 | 参考基线 |
|------|----------|----------|----------|
| 1 | 凭据管理 | 所有密码/密钥迁移至环境变量或 K8s Secret，移除 docker-compose 默认值 | 12-Factor App 配置原则 |
| 2 | 安全上下文 | 所有 K8s Deployment 添加 securityContext（`runAsNonRoot`、`readOnlyRootFilesystem`、drop ALL capabilities） | K8s Pod Security Standards Restricted |
| 3 | 网络策略 | 添加 NetworkPolicy 限制 Pod 间流量，添加 PDB 保障最小可用副本 | K8s 网络隔离最佳实践 |
| 4 | 监控覆盖 | Prometheus 抓取配置覆盖所有核心服务，Alertmanager 正确连接 | Prometheus 监控规范 |
| 5 | CI/CD 门禁 | 移除 `continue-on-error`，添加安全扫描（Trivy/gitleaks/npm audit），覆盖率阈值门禁 | DevSecOps 流水线规范 |

## 五、进度监控与风险应对

### 5.1 进度监控机制

| 机制 | 频次 | 内容 | 责任方 |
|------|------|------|--------|
| 日站会 | 每日 15 分钟 | 同步修复进度，识别阻塞项 | 各修复小组 |
| 周度报告 | 每周一次 | 生成修复进度报告，统计已完成/进行中/阻塞的问题数 | 项目经理 |
| 里程碑评审 | 每个里程碑到达时 | 进行评审，确认达标后方可进入下一阶段 | 项目经理 + 技术负责人 |
| 看板跟踪 | 实时 | 使用 Issue Tracker 看板跟踪每项问题的状态（待开始/进行中/待验证/已关闭） | 全体成员 |

### 5.2 风险应对机制

| 风险点 | 概率 | 影响 | 应对策略 |
|--------|------|------|----------|
| 凭据轮换导致服务中断 | 中 | 高 | 在低峰时段操作，准备回滚方案，轮换后立即验证所有服务连通性 |
| 用户状态逻辑修复影响存量用户 | 中 | 高 | 修复前导出存量用户状态分布，修复后批量校验，准备数据修正脚本 |
| 支付 merchantId 修复影响在途交易 | 低 | 极高 | 修复前排查所有在途支付记录，修复后对账验证，准备资金修正方案 |
| K8s securityContext 导致容器启动失败 | 中 | 中 | 先在 staging 环境验证，准备快速回滚的 Deployment 配置 |
| CI/CD 门禁生效阻塞现有 PR | 高 | 中 | 分阶段启用门禁（先告警后阻断），设置过渡期允许手动覆盖 |
| 前端类型重构引入新 bug | 中 | 中 | 重构前后运行 E2E 测试验证，保持 API 响应结构不变 |
| AI 推理功能未实现影响验收 | 高 | 高 | 验收标准中明确 AI 推理为独立里程碑，不纳入本次修复范围 |
